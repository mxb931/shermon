Quick Start

SherMon is easiest to use as a short workflow: confirm the live connection, review the Alerts page, drill into the affected component, acknowledge if needed, and use the Admin tools when you need to send test traffic, adjust runtime settings, or inspect logs.

1. Confirm the dashboard is live

  • Check the connection badge in the upper-right corner of the header.
  • connected means the page is receiving live websocket updates.
  • reconnecting means the browser is trying to restore the live feed.
  • stale means the page is not currently receiving live updates.

2. Use the Alerts page as the main operating view

  • The summary chips at the top show how many components are currently OK, WARN, CRITICAL, STALE, or DISABLED.
  • The main grid starts at the store level. Select a store to switch the grid to that store's components.
  • Select a component to open its detail view, where you can review current active alerts and the last 24 hours of events for that component.
  • The Incident Stream on the right shows recent unacknowledged events in newest-first order.

3. Review and acknowledge alerts

  • You can acknowledge an alert from the main Incident Stream or from a component's Active Alerts view.
  • An acknowledgement hides the alert from the main Incident Stream, but it does not delete the original event.
  • Use the Acknowledgements page to see all active acknowledgements, who created them, and when they expire.
  • From the Acknowledgements page, you can expire an acknowledgement immediately if you want the alert visible again.

4. Use Test Sender for safe validation

  • Use Mode = Event to create test monitoring traffic.
  • Choose a store and component, then select the event type and a valid severity for that type.
  • Use stale_interval only when you want to test stale behavior. Leave it blank to avoid arming a stale timeout.
  • Use Mode = Acknowledgement to create a test acknowledgement for an existing event.
  • The result dialog shows both the payload that was sent and the API response returned by the backend.

5. Use Admin Config carefully

  • Admin Config changes live runtime settings.
  • Sweeper Interval changes how often stale timeouts and acknowledgement expiry checks run.
  • Entity History Default Rows and Entity History Row Options change how much event history is shown in component detail views.
  • Max Log File Size and Retained Log Files control backend log rotation behavior.
  • Use Reload to refresh current settings and Reset To Defaults only when you want to restore the standard values immediately.

6. Use the Logs page for troubleshooting

  • The Logs page can filter by file, time range, severity, message type, source, state, event id, client IP, and free-text search.
  • Live View refreshes the results every 2 seconds and keeps the results pinned to the newest rows.
  • Prev and Next page through results when Live View is off.
  • If you need to inspect one specific event or client action, filtering by event_id, message_type, or client_ip is usually the fastest path.

7. A simple operator workflow

  • Open Alerts and confirm the connection badge is connected.
  • Use summary chips or the store grid to find affected stores and components.
  • Open the component detail view to see active alerts and recent event history.
  • Acknowledge an alert if it is being worked and should drop out of the main Incident Stream.
  • Use Logs if you need backend detail, or Test Sender if you need to verify expected UI behavior with controlled input.

Field Glossary

  • store_id/component: identify the monitored entity.
  • event_type: problem, ok, disable, or enable.
  • severity: validated by event type (problem=warning/critical; ok=info; disable=info; enable=info).
  • dedup_key: groups repeated reports for the same issue.
  • stale_interval: optional event timing duration in d/h/m format that sets stale timeout behavior.
  • expires_at: acknowledgement expiry timestamp (acks only).
  • metadata: in Test Sender, enter custom metadata as key=value lines (one per line).

Alert Levels

  • CRITICAL (Red): critical issue needing immediate action.
  • WARN (Yellow): warning condition requiring awareness.
  • STALE (Purple): heartbeat timeout (missed expected green check-in).
  • OK (Green): healthy and functional.
  • DISABLED (White): monitor disabled; timeout and alerting suppressed.

Status Hierarchy Rules

  • Store buttons use the highest active component level in this order: CRITICAL, WARN, STALE, OK, DISABLED.
  • Component buttons use the component's current status color using the same priority labels.
  • Only CRITICAL, WARN, and STALE states are active alerts. OK and DISABLED are not active alerts.
  • If a green/OK signal is received while a component is red, yellow, or purple, SherMon clears existing active alerts for that component and resets it to OK.

Acknowledgements

You can acknowledge alerts from Incident Stream or from a component's Active Alerts view. Acknowledged active alerts remain visible in component details with an acknowledged indicator until acknowledgement expiry.

Troubleshooting

  • Connection badge stale/reconnecting indicates websocket interruption.
  • 401 errors indicate API key mismatch.
  • 422 errors indicate invalid field values or expired timestamps.
