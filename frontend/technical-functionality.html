How The Stale Sweeper Works

Stale monitoring is what turns a component purple when an expected check-in does not arrive on time. From an operator perspective, the important point is that stale behavior is controlled by incoming events and then enforced by a background check.

1. Event ingest arms or clears stale monitoring

  • Every ingested event updates the entity status row for that store/component.
  • last_checkin_at is always set to the event time.
  • If the event includes stale_interval, SherMon sets a timeout expectation for the next check-in.
  • If the event omits stale_interval, SherMon clears that timeout expectation.
  • If the component is disabled, stale monitoring is also forced off.

That means the intended behavior is one-shot:

  • Event with stale_interval: arm one timeout window.
  • Later event without stale_interval: clear that timeout window.
  • Later event with a new stale_interval: replace the old timeout window.

2. What the sweeper checks

A background sweeper checks only rows where all of these are true:

  • The component is not disabled.
  • stale_interval_seconds is not null.
  • last_checkin_at is not null.

For each matching row, SherMon compares the current time to the last check-in time.

  • If elapsed time is less than or equal to the stored stale interval, nothing happens.
  • If the status is already purple, nothing happens.
  • Otherwise, the status changes to purple.

When that happens, SherMon updates:

  • status_color = purple
  • last_changed_at = now
  • last_message = Heartbeat timeout: expected green check-in was missed

3. When the sweeper runs

The sweeper runs in the backend on a repeating interval. By default it checks once per minute, applies stale and acknowledgement expiry rules, then sends any updates to connected browsers.

That means a stale interval of 1m is evaluated on the next sweeper pass after that minute has elapsed, not by a dedicated timer for each component.

4. How the UI turns purple

When the sweeper marks a component stale, connected dashboards receive the update and the component turns purple on screen without requiring a manual refresh.

5. What the sweeper does not do

  • It does not create a new incident event row.
  • It does not increment active incident count.
  • It does not re-arm stale monitoring by itself.
  • It only reads the current entity status row and reacts to whatever stale expectation is currently stored there.

6. Expected one-shot behavior

If you send an event with stale_interval, wait until it turns purple, and then send a later event for the same store/component with no stale_interval, that later event should clear stale monitoring and return the component to normal event-driven behavior.

In other words: if stale_interval_seconds is null after the later event, the sweeper should skip that row and it should not turn purple again unless a future event re-arms stale monitoring.

Event Ingest And Status Projection

Every accepted event can affect what operators see on the dashboard. In practice, SherMon keeps both a running current status for each component and a separate event history for audit and troubleshooting.

1. Input validation and normalization

  • The API accepts four event types: problem, ok, disable, and enable.
  • Severity must match the event type so that invalid combinations are rejected early.
  • Older senders that still use recovery are treated the same as ok.
  • stale_interval uses d/h/m duration segments such as 2d5h10m.

2. Event idempotency and active-alert deduplication

  • If an event_id already exists, SherMon treats it as the same event and does not create a duplicate record.
  • If a new problem arrives for the same store/component with the same dedup_key while that issue is already active, the extra report is kept in history but does not increase the active count again.
  • This helps keep the dashboard counts stable even when the same issue is reported more than once.

3. How status color is assigned

  • disable maps to white.
  • enable and ok map to green.
  • problem severity critical maps to red.
  • problem severity warning maps to yellow.

4. When active incidents open or close

  • Only red, yellow, and purple are treated as active alert colors.
  • A problem event opens an active incident only when it is not a duplicate of an already active issue.
  • An ok event closes matching active rows for the same entity and dedup key.
  • A disable event closes all active rows for the entity and resets active incident count to zero.
  • A green event received while the current status is red, yellow, or purple performs a full active-alert reset for that entity.

5. What is saved for each accepted event

  • A historical event row is saved in incident_events.
  • The current component state is updated in entity_status.
  • The system sequence counter advances so realtime consumers can stay in order.
  • The outbound realtime payload is also saved so the stream has durable history.

Bootstrap And Realtime Synchronization

The dashboard is built to load quickly and then stay current. Operators first get a complete snapshot of the current state, then the browser listens for live changes.

1. Bootstrap snapshot

  • /api/v1/bootstrap returns current statuses, recent events, active acknowledgements, and runtime settings.
  • The page uses that response to build the summary chips, store/component grid, and incident stream before live updates start arriving.
  • This is why a newly opened browser tab can show the current state immediately.

2. Websocket steady-state updates

  • The browser connects to /ws/updates and keeps that connection alive with periodic pings.
  • The backend pushes event updates, stale changes, acknowledgement updates, and acknowledgement expiry messages.
  • The page updates only what changed, then redraws the affected views.

3. Connection handling

  • The connection badge cycles through reconnecting, connected, and stale.
  • If the websocket closes, the browser backs off exponentially up to 15 seconds before retrying.
  • The server keeps track of active websocket clients and removes connections that can no longer receive updates.

Acknowledgement Lifecycle

Acknowledgements help operators reduce noise while keeping the original event history intact. An ack changes how an alert is presented, not whether the event happened.

1. Creating or updating an acknowledgement

  • You can only acknowledge an event that already exists.
  • If the same event is acknowledged again, SherMon updates the existing acknowledgement instead of creating a second one.
  • The main incident stream hides events that currently have an active acknowledgement.

2. Expiry behavior

  • Acknowledgements can expire manually or automatically.
  • When an acknowledgement expires, it becomes inactive and the browser is notified.
  • The original event remains in history and can become visible again once the acknowledgement is gone.

3. Why acknowledgements are handled separately

  • Event history stays intact for audit and troubleshooting.
  • Acknowledgement timing can change without altering the original event.
  • The same event can move between visible and acknowledged states without being deleted.

Runtime Configuration And Log Access

SherMon includes a small set of runtime controls so operators can tune monitoring behavior and review backend activity without editing code or restarting for every change.

1. Runtime configuration model

  • sweeper_interval_seconds controls how often stale and ack expiry checks run.
  • entity_history_default_limit and entity_history_limit_options control how many historical rows the component detail view offers by default.
  • log_max_mb and log_backup_count control backend log rotation.

2. Runtime config application path

  • The config is stored as a single shared runtime settings row.
  • When config is updated through the API, logging settings are reapplied immediately.
  • The sweeper reads the current interval on each cycle, so cadence changes take effect without restarting the service.

3. Log browser behavior

  • The backend lists files in the configured log directory that match the active log file prefix.
  • Each log line is parsed into fields such as message_type, source, state, and event_id.
  • The UI can filter logs by time range, severity, source, event id, client IP, and free-text query.

Startup, Schema Compatibility, And Migration Safety

SherMon performs startup checks to keep the database aligned with the application. This reduces upgrade friction and helps existing data remain usable as the application evolves.

1. Table creation and compatibility checks

  • Missing tables are created automatically.
  • Startup checks add newer columns when they are missing.
  • Compatibility helpers backfill newer fields from older data when possible.

2. Legacy cleanup behavior

  • Legacy recovery events are rewritten to ok for consistency.
  • Older schema fields that are no longer used can be cleaned up during startup, with a safe fallback for SQLite environments that have limited schema-change support.

3. Why this matters operationally

  • Upgrades are more forgiving in development and small deployments.
  • Existing data can be preserved while newer application behavior is adopted.
  • Startup becomes the main safeguard for keeping persistent data aligned with current expectations.

Frontend Hierarchy And Rendering Flow

The Alerts page is designed to stay responsive by keeping current state in the browser and re-rendering from that local state as updates arrive.

1. Store and component hierarchy

  • The top-level grid groups components by store and shows each store using the highest-priority component color.
  • Selecting a store reuses the same grid area to show components for that store.
  • Summary chips use the same current-status data to open modal views for each color bucket.

2. Incident stream rendering

  • The incident stream stays in newest-first order.
  • The list is capped so the page does not grow without limit over time.
  • Acknowledged events are hidden from the main stream, but remain available through entity-specific views and historical event queries.

3. Entity detail modal behavior

  • Opening a component detail view shows both currently active alerts and the last 24 hours of event history.
  • The event history row limit follows runtime config.
  • Metadata is shown in expandable sections so operators can inspect detail only when needed.
