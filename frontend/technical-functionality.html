How The Stale Sweeper Works

The stale monitor has two parts: event ingest arms or clears the timeout expectation, and a background sweeper periodically checks whether that expectation was missed.

1. Event ingest arms or clears stale monitoring

  • Every ingested event updates the entity status row for that store/component.
  • last_checkin_at is always set to the event time.
  • If the event includes stale_interval, SherMon parses it and stores it as stale_interval_seconds.
  • If the event omits stale_interval, SherMon explicitly clears the stale expectation by setting stale_interval_seconds to null.
  • If the component is disabled, stale monitoring is also forced off.

That means the intended behavior is one-shot:

  • Event with stale_interval: arm one timeout window.
  • Later event without stale_interval: clear that timeout window.
  • Later event with a new stale_interval: replace the old timeout window.

2. What the sweeper checks

A background sweeper scans entity status rows and only evaluates rows where all of these are true:

  • The component is not disabled.
  • stale_interval_seconds is not null.
  • last_checkin_at is not null.

For each matching row, SherMon computes elapsed time as now - last_checkin_at.

  • If elapsed time is less than or equal to the stored stale interval, nothing happens.
  • If the status is already purple, nothing happens.
  • Otherwise, the status changes to purple.

When that happens, SherMon updates:

  • status_color = purple
  • last_changed_at = now
  • last_message = Heartbeat timeout: expected green check-in was missed

3. When the sweeper runs

The sweeper runs in the backend on a 60-second loop. It sleeps for 60 seconds, opens a database session, runs timeout checks and acknowledgement expiry checks, then broadcasts any resulting realtime updates.

That means a stale interval of 1m is evaluated on the next sweeper pass after that minute has elapsed, not by a per-entity timer.

4. How the UI turns purple

When the sweeper marks a component stale, it emits a websocket message of kind status_timeout. The frontend receives that payload, updates the in-memory status for that store/component, and re-renders the dashboard.

5. What the sweeper does not do

  • It does not create a new incident event row.
  • It does not increment active incident count.
  • It does not re-arm stale monitoring by itself.
  • It only reads the current entity status row and reacts to whatever stale expectation is currently stored there.

6. Expected one-shot behavior

If you send an event with stale_interval, wait until it turns purple, and then send a later event for the same store/component with no stale_interval, the next event should clear stale monitoring and return the component to normal event-driven behavior.

In other words: if stale_interval_seconds is null after the later event, the sweeper should skip that row and it should not turn purple again unless a future event re-arms stale monitoring.
