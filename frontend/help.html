Help

Quick Start

Use Alerts to navigate a hierarchy: Stores -> Components -> Active Alerts. Select a store to drill into its components, then select a component to review and acknowledge active alerts. Use Test Sender to generate valid test traffic.

Field Glossary

  • store_id/component: identify the monitored entity.
  • event_type: problem, ok, disable, or enable.
  • severity: validated by event type (problem=warning/critical; ok=info; disable=info; enable=info).
  • dedup_key: groups repeated reports for the same issue.
  • stale_interval: optional event timing duration in d/h/m format that sets stale timeout behavior.
  • expires_at: acknowledgement expiry timestamp (acks only).
  • metadata: in Test Sender, enter custom metadata as key=value lines (one per line).

Alert Levels

  • CRITICAL (Red): critical issue needing immediate action.
  • WARN (Yellow): warning condition requiring awareness.
  • STALE (Purple): heartbeat timeout (missed expected green check-in).
  • OK (Green): healthy and functional.
  • DISABLED (White): monitor disabled; timeout and alerting suppressed.

Status Hierarchy Rules

  • Store buttons use the highest active component level in this order: CRITICAL, WARN, STALE, OK, DISABLED.
  • Component buttons use the component's current status color using the same priority labels.
  • Only CRITICAL, WARN, and STALE states are active alerts. OK and DISABLED are not active alerts.
  • If a green/OK signal is received while a component is red, yellow, or purple, SherMon clears existing active alerts for that component and resets it to OK.

Acknowledgements

You can acknowledge alerts from Incident Stream or from a component's Active Alerts view. Acknowledged active alerts remain visible in component details with an acknowledged indicator until acknowledgement expiry.

Troubleshooting

  • Connection badge stale/reconnecting indicates websocket interruption.
  • 401 errors indicate API key mismatch.
  • 422 errors indicate invalid field values or expired timestamps.
